## Design Context

### Users

- **Primary audience**: Recruiters evaluating technical talent, potential clients exploring partnership, technical peers assessing credibility
- **Context**: Arriving from LinkedIn, GitHub, search, or direct referral — typically forming a judgment within 10 seconds
- **Job to be done**: Quickly understand who Juan Almeida is, what he builds (AI systems, automations, full-stack applications), and whether to engage further (hire, collaborate, connect)
- **Secondary audience**: Technical peers and open-source collaborators exploring work history, skill depth, and published projects
- **Locales**: English (primary) and Spanish — all UI text must be bilingual with identical key structures

### Brand Personality

- **Voice**: Developer-native, technically precise, confident without arrogance
- **Tone**: Precise but playful — system prompts, TODO metaphors, and command-like labels feel native to engineering, not gimmicky
- **3-word personality**: **Precise · Playful · High-Agency**
- **Emotional goals** (priority order):
  1. **Curiosity** — "I want to explore more"
  2. **Confidence** — "This person knows what they're doing"
  3. **Delight** — "This is unexpectedly fun"
  4. Trust — "I can rely on this person"
  5. Inspiration — "I want to build something like this"
- **Emotional arc across the page**:
  - First impression: curious, sharp, unconventional
  - Mid-scroll: competent, multifaceted, credible
  - End state: confident that Juan can design, build, and ship technically sophisticated products

### Aesthetic Direction

- **Visual tone**: Editorial developer desktop — cinematic in transitions, restrained in layout, technical without coldness
- **Core concept**: The page behaves like a personal execution board. Visitors experience a sequence they complete with you, not a brochure they skim. A persistent TODO rail drives navigation and progress. Task labels use playful developer language: _Boot identity_, _Compile strengths_, _Unlock work log_, _Open channel_.
- **Mood descriptors**: Precise but playful · High-agency · Technical without coldness · Cinematic in transitions, restrained in layout

#### Theme System

Two named themes with distinct character, not just inverted colors:

| Role                    | Build Mode (light)                                                           | After Hours (dark)                                            |
| ----------------------- | ---------------------------------------------------------------------------- | ------------------------------------------------------------- |
| **Surfaces**            | Chalk, concrete, pale graphite (not pure white)                              | Charcoal, ink, smoked teal, near-black                        |
| **Primary signal**      | Acid green / digital lime (~#00FF80) — completion, focus, active rail states | Same hue, adjusted for dark-surface contrast                  |
| **Secondary signal**    | Electric cyan — hover trails, info accents, data annotations                 | Same role, brighter on dark surfaces                          |
| **Warm counter-accent** | Muted amber — warnings, timestamps, detail moments                           | Same role, warmer glow on dark surfaces                       |
| **Portrait character**  | Calm, confident, approachable, attentive                                     | Sharper, more playful, sunglasses reveal as personality shift |

Theme names ("Build Mode" / "After Hours") are more expressive than plain "Light" / "Dark." The toggle triggers a choreographed transition: surfaces, accent roles, and portrait imagery change in one coordinated moment under 650ms.

#### Color Accessibility

- Checked/active states never rely on color alone — always paired with icon, stroke, or text-decoration change
- Active section state requires at least 4.5:1 contrast ratio for text and a clearly visible focus ring
- Portrait overlays must not reduce adjacent text contrast below WCAG AA thresholds

#### Typography

Monospace-forward display layer paired with a human editorial sans for long-form copy:

| Level | Purpose                        | Desktop Size | Mobile Size | Font           |
| ----- | ------------------------------ | ------------ | ----------- | -------------- |
| 0     | Hero display (Hello World)     | 64–88px      | 32–40px     | JetBrains Mono |
| 1     | Section titles                 | 36–48px      | 26–30px     | JetBrains Mono |
| 2     | Support labels, rail states    | 14–18px      | 14–16px     | JetBrains Mono |
| Body  | Descriptive copy               | 18–20px      | 16–18px     | Inter          |
| Meta  | Timestamps, tags, state labels | 13–15px      | 12–14px     | JetBrains Mono |

- Heavy weights used sparingly for key claims and TODO completion states
- Descriptive paragraphs stay at regular or medium to prevent visual fatigue
- Responsive sizing uses `clamp()` values (defined in Phase 1)

#### Photography

Editorial portraits are a core design material, not filler:

- **Hero portrait pair**: One base + one sunglasses variant for theme transformation (4–8 frame sequence on toggle)
- **Supporting editorial crops**: 2–3 distributed across Intro, Skills, and Work sections
- **Detail crop**: Hands, face crop, or desk interaction shot for texture in smaller modules
- **Shot guidance**: Clean studio-style for hero; environmental workspace portrait for bio/work chapter; tight editorial crop for transitions
- **Frame sequence rules**: All hero frames identical dimensions/crop/alignment; WebP/AVIF assets with non-animated fallback for reduced motion

#### Composition

- **Desktop**: Split composition — large content stage left, persistent TODO rail right
- **Tablet**: Narrower sticky rail column with shorter labels or icon+label format
- **Mobile**: Sticky top progress strip or bottom task dock with expandable checklist
- **Section rhythm**: Alternate density (text-led → image-led → card-led) to avoid repetitive scanning
- **Whitespace**: Generous vertical spacing around hero and work sections; tight spacing for system labels, stats, metadata
- **Asymmetry**: Portrait crops can break grid edge; occasional overlaps between photo planes and copy containers — limited to one emphasis move per section

#### Icons

- Material Symbols Outlined (Google Fonts) — primary icon system
- Font Awesome (`@fortawesome/fontawesome-free`) — available for supplemental use

#### Anti-References

- Generic developer portfolio templates (free Astro/Next themes)
- Corporate SaaS landing pages
- Futuristic particle/neon portfolio clichés with random particles, vague neon noise, or motion that obscures content
- Stock-photo confidence gestures or exaggerated expressions
- "Template look" — the experience must feel authored and differentiated

### Design Principles

1. **Narrative over brochure** — Every section progresses through a meaningful sequence. The TODO rail drives orientation, not a traditional navbar. Visitors watch work items resolve as the story unfolds.

2. **Developer culture as native metaphor** — Hello World, TODO states, command-like labels, and completion logic feel like engineering, not decoration. The metaphor is the interface, not a skin on top of it.

3. **Motion confirms progress** — Every animation answers one of three questions: _Where am I now?_, _What changed?_, or _Why should I care?_ No decorative motion without purpose. Chapter-specific reveal families (hero types in, intro wipes on, skills assemble, work cascades, contact resolves).

4. **Photography carries personality** — Portraits are authored editorial elements integrated into the design system, with theme-aware variants and frame-by-frame transitions. They make the interface feel personal and unmistakable.

5. **Machine-readable always** — Semantic HTML, source-order clarity, and content visibility survive even if JavaScript fails or effects are disabled. WCAG AA minimum. `prefers-reduced-motion: reduce` replaces animations with instant state changes plus short opacity fades. Spanish locale expansion is handled without rigid widths that break on translation.

6. **Reward exploration with layered detail** — Fast scanners get the pitch quickly. Deeper visitors discover richer transitions, hover states, and content reveals. One high-attention animation active per viewport region at a time.

7. **Right rail is sacred** — The TODO navigation is the main orientation system. It stays readable, useful, and responsive across all breakpoints. It teaches the interaction model within the first seconds by resolving the first task visibly.

### Motion Language

- **Page load**: TODO rail appears first (incomplete checklist) → Hello World types/resolves → portrait enters via masked reveal → first TODO item cycles through pending → active → completed
- **Section reveals**: Each chapter has a distinct reveal family — no uniform fade-ins
- **Hover/focus**: Directional motion tied to pointer entry; strong keyboard-visible focus states
- **Scroll-linked**: Hero can pin briefly during first TODO resolution; portrait layers shift subtly with scroll (opacity + translate only); work section cards cascade with scroll progress
- **Reduced motion**: All transforms replaced with instant state + short opacity transitions; checklist logic and theme semantics preserved
- **Performance budget**: Prefer `opacity`, `transform`, limited `clip-path`, and CSS variable transitions. Use `IntersectionObserver` over scroll handlers. One signature transition per chapter maximum.

### TODO Rail Labels

The rail labels use playful developer-language rather than plain imperative labels:

| Section         | Label             |
| --------------- | ----------------- |
| Hero / Intro    | Boot identity     |
| Bio / Skills    | Compile strengths |
| Projects / Work | Unlock work log   |
| Contact         | Open channel      |

Three-state logic per item: **pending** → **active** (cursor sweep) → **completed** (checkmark draw + strike-through settle). The first item resolves within the opening interaction to teach the model.
